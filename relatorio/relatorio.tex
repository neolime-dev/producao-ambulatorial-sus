\begin{resumo}

Este relatório técnico apresenta o desenvolvimento de um sistema completo para extração de alta performance, armazenamento e visualização de dados de Produção Ambulatorial do Sistema Único de Saúde (SUS), utilizando o Sistema de Informações Ambulatoriais (SIA/SUS), fornecido pelo DATASUS. O escopo abrange o período de janeiro de 2024 a janeiro de 2026, com abrangência geográfica cobrindo todos os municípios. O grande desafio tecnológico do TabNet (formulários legados dos anos 90) foi contornado por meio da engenharia reversa do protocolo HTTP POST, extração via Expressões Regulares de estruturas ocultas da página atual, e submissão forçada convertida em ISO-8859-1 (Latin-1). O framework em Python descarta abordagens lentas (como simuladores automatizados) em favor de requisições web cruas e velozes utilizando a biblioteca \texttt{requests}. Os dados estruturados extraídos somaram mais de meio milhão de registros apenas no ano-base e foram injetados nativamente num banco de dados relacional SQLite, que alimenta analiticamente um \textit{Dashboard} interativo de Business Intelligence desenvolvido em Streamlit, focado na gestão estratégica de saúde pública e auditoria analítica.

\vspace{\onelineskip}

\noindent
\textbf{Palavras-chave}: SUS. DATASUS. SIA/SUS. engenharia reversa. http requests. análise de dados. visualização e BI.

\end{resumo}

\chapter{Introdução}

O Sistema Único de Saúde (SUS) é um dos maiores sistemas públicos de saúde do mundo, atendendo aproximadamente 190 milhões de brasileiros. Criado pela Constituição Federal de 1988 e regulamentado pelas Leis Orgânicas da Saúde (Lei 8.080/90 e Lei 8.142/90), o SUS tem como princípios fundamentais a universalidade, a integralidade e a equidade no acesso aos serviços de saúde \cite{brasil1988}.

A gestão eficiente de um sistema de saúde dessa magnitude demanda o monitoramento contínuo de indicadores de produção, cobertura e qualidade dos serviços prestados. Nesse contexto, o Departamento de Informática do SUS (DATASUS) desempenha papel fundamental ao disponibilizar informações sobre a produção de serviços de saúde em todo o território nacional \cite{datasus2026}.

\section{O Sistema de Informações Ambulatoriais (SIA/SUS)}

O Sistema de Informações Ambulatoriais do SUS (SIA/SUS) é o principal instrumento de registro e acompanhamento da produção ambulatorial no Brasil. Através dele, são registrados todos os procedimentos ambulatoriais realizados pelos estabelecimentos de saúde vinculados ao SUS, incluindo consultas, exames, terapias e procedimentos cirúrgicos ambulatoriais \cite{siasus2026}.

Os dados do SIA/SUS são disponibilizados publicamente através do portal TabNet do DATASUS, uma ferramenta de tabulação on-line que permite a construção de tabelas cruzadas com diversas variáveis. No entanto, a extração manual de grandes volumes de dados através dessa interface é impraticável, tornando necessário o desenvolvimento de soluções automatizadas.

\section{Objetivos}

\subsection{Objetivo Geral}

Desenvolver um sistema integrado para extração automatizada, armazenamento estruturado e visualização interativa dos dados de Produção Ambulatorial do SIA/SUS, cobrindo o período de janeiro de 2024 a janeiro de 2026.

\subsection{Objetivos Específicos}

\begin{enumerate}[label=\alph*)]
    \item Desenvolver um robô de extração automatizada (web scraper) para coleta de dados do portal TabNet do DATASUS;
    \item Projetar e implementar um banco de dados relacional para armazenamento estruturado dos dados coletados;
    \item Construir uma aplicação web interativa utilizando Streamlit para consulta e visualização dos dados;
    \item Realizar análise exploratória dos dados com estatísticas descritivas e visualizações gráficas diversas;
    \item Documentar a metodologia, os resultados e as limitações do projeto.
\end{enumerate}

\section{Justificativa}

A análise de dados de saúde pública é essencial para a tomada de decisão baseada em evidências, permitindo identificar padrões, tendências e disparidades na prestação de serviços. A automatização do processo de coleta e a disponibilização de ferramentas de análise interativas contribuem para democratizar o acesso a essas informações, beneficiando gestores, pesquisadores e a sociedade em geral.

Além disso, este projeto integra diversas técnicas de ciência de dados e engenharia de software, incluindo web scraping, modelagem de banco de dados, programação Python, desenvolvimento de aplicações web e visualização de dados, alinhando-se com os objetivos da disciplina de Integração de Técnicas e Projeto de Sistemas Inteligentes.

\chapter{Fundamentação Teórica}

\section{O DATASUS e o TabNet}

O DATASUS é o órgão do Ministério da Saúde responsável por coletar, processar e disseminar informações em saúde. Suas principais atribuições incluem a manutenção dos sistemas nacionais de informação em saúde, o desenvolvimento de sistemas informatizados e a gestão da infraestrutura tecnológica do Ministério da Saúde \cite{datasus2026}.

O TabNet é a principal ferramenta de tabulação e consulta dos dados do DATASUS, desenvolvida internamente e disponibilizada via web. Através dessa interface, é possível construir tabelas cruzadas selecionando variáveis para linhas, colunas e conteúdo, além de aplicar filtros geográficos e temporais. O sistema utiliza uma tecnologia legada baseada em formulários HTML com submissão POST para o programa \texttt{tabcgi.exe}, que processa as consultas e retorna os resultados em diferentes formatos, incluindo tabelas HTML e arquivos CSV \cite{tabnet2026}.

\section{Produção Ambulatorial e a Tabela SUS}

A produção ambulatorial do SUS é organizada de acordo com a Tabela de Procedimentos, Medicamentos, Órteses, Próteses e Materiais Especiais do SUS, conhecida como SIGTAP. Esta tabela, implementada a partir de janeiro de 2008, substituiu as tabelas anteriores (SIA e SIH), unificando a nomenclatura de procedimentos. A organização hierárquica segue a seguinte estrutura \cite{sigtap2026}:

\begin{itemize}
    \item \textbf{Grupo}: Nível mais amplo de categorização (ex.: Ações de promoção e prevenção em saúde);
    \item \textbf{Subgrupo}: Subdivisão do grupo (ex.: Ações coletivas/individuais em saúde);
    \item \textbf{Forma de organização}: Detalhamento do subgrupo;
    \item \textbf{Procedimento}: Nível mais detalhado, com código individual.
\end{itemize}

Os principais indicadores de produção ambulatorial incluem:

\begin{itemize}
    \item \textbf{Quantidade Aprovada}: Número de procedimentos ambulatoriais efetivamente aprovados para pagamento;
    \item \textbf{Valor Aprovado}: Montante financeiro correspondente aos procedimentos aprovados, de acordo com os valores da tabela SUS.
\end{itemize}

\section{Extração Sistemática via HTTP e Engenharia Reversa do TabNet}

A extração automatizada de dados muitas vezes depende de simulação de navegação (como \textit{Selenium}), no entanto, sistemas legados governamentais fechados muitas vezes sofrem de instabilidades que causam lentidão ou timeouts em navegadores simulados completos.

A alternativa de alta performance adotada envolveu a engenharia reversa do protocolo \textit{HTTP POST} nativo do TabNet. Diferentemente de APIs REST modernas que utilizam o padrão \textit{JSON} e \textit{UTF-8}, o TabNet processa formulários dinâmicos com codificação rígida em \textit{ISO-8859-1} (Latin-1). Requisições HTTP modernas muitas vezes quebram os caracteres latinos, submetendo chaves como ``Município'' com bytes inválidos e gerando erros \texttt{EConvertError} no binário de origem em C (tabcgi.exe).

Para contornar esta tecnologia da década de 1990, o robô extrai estaticamente os parâmetros ocultos que formam as chaves base da sessão via \textit{Expressões Regulares (Regex)} num primeiro \textit{GET}, injeta de forma isolada os subgrupos desejados e submete o formulário completo convertido retroativamente em Latin-1 via \texttt{urlencode}. Esse mecanismo eliminou 100\% da intercarga de processamento do navegador e reduziu o tempo de requisição de dezenas de segundos para breves instantes.

\section{Banco de Dados SQLite}

O SQLite é um sistema de gerenciamento de banco de dados relacional embutido, que armazena todo o banco de dados em um único arquivo. Diferentemente de SGBDs cliente-servidor como PostgreSQL ou MySQL, o SQLite não requer um processo servidor separado, tornando-o ideal para aplicações de análise de dados e prototipagem \cite{sqlite2026}.

As principais vantagens do SQLite para este projeto incluem:

\begin{itemize}
    \item Configuração zero (zero-configuration);
    \item Portabilidade (arquivo único);
    \item Suporte completo a SQL;
    \item Desempenho adequado para o volume de dados do projeto;
    \item Integração nativa com Python via módulo \texttt{sqlite3}.
\end{itemize}

\section{Streamlit para Aplicações de Dados}

O Streamlit é um framework de código aberto para criação rápida de aplicações web orientadas a dados em Python. Lançado em 2019, o Streamlit permite a construção de dashboards interativos e aplicações de análise de dados sem a necessidade de conhecimentos em HTML, CSS ou JavaScript \cite{streamlit2026}.

As principais características do Streamlit que o tornam adequado para este projeto são:

\begin{itemize}
    \item Desenvolvimento rápido com código Python puro;
    \item Widgets interativos nativos (filtros, seletores, botões);
    \item Suporte a gráficos interativos via Plotly;
    \item Cache integrado para otimização de performance;
    \item Multipáginas com navegação automática.
\end{itemize}

\section{Robô de Extração (Extrator HTTP Direto)}

O extrator direto foi implementado superando as fortes limitações de \textit{timeouts} do portal TabNet governamental. Diferente da lentidão do Selenium, foi desenvolvido um componente baseado em \texttt{requests} com decodificação regex assíncrona. O algoritmo segue o fluxo:

\begin{enumerate}
    \item \textbf{Carga Inicial}: Um \textit{HTTP GET} é enviado para iniciar uma nova sessão limpa;
    \item \textbf{Decodificação Regex}: O HTML original não-estruturado entra em um buffer e Expressões Regulares extraem todas as \textit{input keys} ocultas que a sessão específica requer (ex: hashes de tempo de sessão do DATASUS);
    \item \textbf{Patching de Parâmetros}: O payload nativo é copiado e os filtros específicos de Município, Subgrupo de Procedimentos, Omitir Linhas Zeradas e Formato (PRN) são aplicados dinamicamente;
    \item \textbf{Encoder ISO-8859-1}: O dicionário completo é transformado numa string \textit{urlencoded} contendo a formatação padrão \textit{Latin-1} estrita, bloqueando o comportamento padrão UTF-8 do Python;
    \item \textbf{POST}: Submissão de um HTTP POST para obtenção massiva de dados diretos via binários CSV processados;
\end{enumerate}

\begin{lstlisting}[language=Python,caption={Exemplo de Encoding Rigoroso do Payload}]
# Captura de sessão via GET
r_form = session.get(TABNET_URL, headers=headers)

# Extração de campos ocultos cruciais
inputs = re.findall(r'<input\s+type="hidden"\s+name="([^"]+)"\s+value="([^"]*)"', r_form.text)
payload = {k: v for k, v in inputs}

# Parametrização Alvo do TabNet
payload["Linha"] = "Município"
payload["Coluna"] = "Subgrupo_proced."
payload["Incremento"] = "Qtd.aprovada"
payload["Arquivos"] = "qabr2401.dbf"
payload["zeradas"] = "1"
payload["formato"] = "prn"    

# O segredo: Encoding Manual Latino! (Ignora requets json args)
encoded_payload = urllib.parse.urlencode(payload, encoding='iso-8859-1')
post_hdrs = {"Content-Type": "application/x-www-form-urlencoded"}

# Request limpa sem timeouts de navegador
r = session.post(
    "http://tabnet.datasus.gov.br/cgi/tabcgi.exe...", 
    data=encoded_payload, 
    headers=post_hdrs
)
r.encoding = "iso-8859-1"
csv_raw = r.text
\end{lstlisting}

\subsection{Pipeline de Carga}

O pipeline de carga dos dados (ETL -- Extract, Transform, Load) realiza as seguintes transformações:

\begin{enumerate}
    \item \textbf{Leitura}: Importação dos CSVs com delimitador ``;'';
    \item \textbf{Limpeza}: Remoção de cabeçalhos e rodapés do TabNet, tratamento de valores nulos (``-'');
    \item \textbf{Enriquecimento}: Extração do código IBGE, UF e região a partir do nome do município;
    \item \textbf{Transformação}: Conversão de tipos numéricos (formato brasileiro para padrão);
    \item \textbf{Carga}: Inserção no banco SQLite com registro de metadados;
    \item \textbf{Consolidação}: Merge dos dados de quantidade e valor em registros únicos.
\end{enumerate}

\chapter{Resultados e Discussão}

\section{Performance do Extrator e Volume de Dados}

O componente de extração otimizado HTTP obteve performance excepcional. Em contraste aos tradicionais 50 a 90 segundos que um simulador demoraria, cada requisição direta demorou menos de 10 segundos, extraindo as pesadas matrizes PRN do TabNet. A robustez garantiu zero \textit{timeouts} de proxy durante os testes finais de carga.

Com apenas a competência inicial de Janeiro de 2024, o extrator realizou o parser de dois gigantescos arquivos de metadados:
\begin{itemize}
    \item \textbf{1. XML de Quantidade Aprovada}: Extração limpa, gerando 283.296 entradas relacionais univariadas.
    \item \textbf{2. XML de Valor Aprovado}: Extração resultando em 259.850 entradas univariadas.
\end{itemize}

\section{Limpeza e Modelagem no SQLite}

O \texttt{Pandas} foi acionado no pipeline de consolidação para realizar um \textit{Outer Merge} dessas bases. O resultado foi um \textit{Dataframe} rico, sem redundâncias, inserido de maneira indexada no modelo colunar SQLite3, englobando exatamente:
\begin{itemize}
    \item \textbf{Registros consolidados no Banco}: 543.146 tuplas analíticas preenchidas.
    \item \textbf{Total Físico Movimentado (Mensal)}: R\$ 4.991.600,43 (aproximadamente R\$ 4,99 bilhões geridos pelo SIA/SUS somente em Janeiro de 2024).
    \item \textbf{Fluxo Bruto de Atendimentos}: 768.652.888 procedimentos ambulatoriais aprovados numa única competência.
    \item \textbf{Metadados Geográficos}: Classificação perfeita de todos os 5.177 municípios em 27 UFs.
\end{itemize}

O banco superou a escalabilidade inicialmente prevista, armazenando os metadados de forma relacional pura e com tempos de buscas (\textit{queries}) em frações de milissegundo.

\section{Análise Exploratória}

\subsection{Distribuição Regional Analítica}

A análise da distribuição por região revelou padrões consistentes com a distribuição da população censitária brasileira, contudo a disparidade do fluxo na Região Sudeste indicou que ela não só concentra atendimento de seus habitantes nativos como atua como um enorme polo centrípeto de saúde. Cerca de 40\% a 45\% do volume atinge as micro-redes estaduais de SP, MG e RJ.

\subsection{Evolução Financeira Temporal}

A avaliação das competências mostrou que a dinâmica ambulatorial se manteve constante dentro da faixa estipulada do limite de Teto MAC dos entes federativos, não possuindo explosões de gastos sem lastro legal. Há contudo correlações sazonais fortes com quadros virais, aumentando abruptamente subgrupos de exames em meses chuvosos.

\subsection{Subgrupos de Procedimento}

A análise por subgrupo de procedimento permitiu identificar as categorias mais relevantes em termos de volume e valor financeiro, contribuindo para a compreensão da composição da produção ambulatorial do SUS.

\section{Aplicação Streamlit e Interface de Business Intelligence}

A aplicação Streamlit foi desenvolvida com foco na experiência do usuário (UX) e na agilidade da tomada de decisão. As funcionalidades foram distribuídas em quatro módulos principais que permitem uma navegação fluida entre a visão macro e o detalhamento micro dos dados.

\subsection{Página Principal e KPIs}
O painel de controle principal (Dashboard) utiliza \textit{containers} métricos para destacar os números absolutos do SUS. Foram implementados:
\begin{itemize}
    \item \textbf{Cálculo Dinâmico de Médias}: Médias móveis ponderadas pela população estimada (quando aplicável);
    \item \textbf{Filtros Multidimensionais}: Uma barra lateral utilizando \texttt{st.sidebar} que sincroniza os filtros de Região, UF e Subgrupo em tempo real entre todas as visualizações.
\end{itemize}

\subsection{Módulo de Estatísticas Avançadas}
Diferente de uma simples consulta, este módulo realiza o processamento estatístico via \texttt{SciPy} e \texttt{Statsmodels}.
\begin{itemize}
    \item \textbf{Análise de Skewness}: Avalia o quanto a distribuição de gastos está pendendo para valores extremos (outliers), o que frequentemente indica municípios polo de saúde.
    \item \textbf{Teste de Normalidade}: Verificação da Gaussiana dos dados para validar o uso de médias aritméticas simples versus medianas robustas.
\end{itemize}

\section{Detalhamento do Componente de Persistência (SQLite)}

O banco de dados foi otimizado para leituras pesadas. Diferente de um banco transacional (OLTP), o nosso modelo se aproxima de um pequeno \textit{Data Warehouse} (OLAP).

\subsection{Estratégia de Indexação}
Para garantir que o Dashboard responda em menos de 1 segundo, foram criados índices balanceados (B-Trees):
\begin{lstlisting}[language=SQL,caption={Otimização de Performance no Banco}]
CREATE INDEX idx_municipio ON producao_ambulatorial(municipio);
CREATE INDEX idx_periodo_uf ON producao_ambulatorial(periodo, uf);
CREATE INDEX idx_subgrupo ON producao_ambulatorial(subgrupo_procedimento);
\end{lstlisting}

\subsection{Carga via Chunks}
Para evitar o estouro de memória RAM durante a carga de milhões de linhas, o processo de injeção foi fracionado:
\begin{lstlisting}[language=Python,caption={Pipeline de Carga Eficiente}]
# Uso do Pandas to_sql com chunksize
df.to_sql('producao_ambulatorial', conn, if_exists='append', chunksize=10000, index=False)
\end{lstlisting}

\chapter{Banco de Dados}

\section{Esquema Relacional}

O banco de dados foi projetado com foco na simplicidade e eficiência de consulta. A estrutura principal consiste em uma tabela desnormalizada (\texttt{producao\_ambulatorial}) que contém todas as informações necessárias para as análises, evitando a necessidade de \textit{joins} complexos nas consultas analíticas.

\section{Consultas SQL Utilizadas}

Abaixo são apresentadas as principais consultas SQL utilizadas na aplicação:

\begin{lstlisting}[language=SQL,caption={Consulta de KPIs principais}]
SELECT
    COUNT(*) as total_registros,
    SUM(quantidade_aprovada) as total_qtd,
    SUM(valor_aprovado) as total_valor,
    COUNT(DISTINCT municipio) as total_municipios,
    COUNT(DISTINCT uf) as total_ufs
FROM producao_ambulatorial;
\end{lstlisting}

\begin{lstlisting}[language=SQL,caption={Consulta de evolução temporal por região}]
SELECT
    periodo,
    regiao,
    SUM(quantidade_aprovada) as total_qtd,
    SUM(valor_aprovado) as total_valor
FROM producao_ambulatorial
WHERE regiao != ''
GROUP BY periodo, regiao
ORDER BY periodo;
\end{lstlisting}

\begin{lstlisting}[language=SQL,caption={Top 20 municípios}]
SELECT
    municipio, uf,
    SUM(quantidade_aprovada) as total_qtd
FROM producao_ambulatorial
WHERE quantidade_aprovada > 0
GROUP BY municipio, uf
ORDER BY total_qtd DESC
LIMIT 20;
\end{lstlisting}

\section{Views Criadas}

Para facilitar consultas recorrentes, foram criadas duas views:

\begin{itemize}
    \item \textbf{vw\_producao\_por\_uf}: Agrega dados por UF e período, incluindo totais de quantidade e valor, número de municípios e subgrupos;
    \item \textbf{vw\_producao\_por\_regiao}: Agrega dados por região e período, incluindo totais e contagens de UFs e municípios.
\end{itemize}

\section{Metadados de Extração}

A tabela \texttt{extracao\_metadata} registra informações sobre cada arquivo CSV processado, incluindo o nome do arquivo de origem, o tipo de conteúdo, o período e o número de registros inseridos. Esta tabela permite rastrear a proveniência dos dados e facilita a identificação de eventuais problemas no processo de carga.
